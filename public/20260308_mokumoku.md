# AWSクラウドコンピューティングの6つのメリット

- 固定費(設備投資費)が柔軟な変動費へ
  今までオンプレで維持していたサーバをクラウドに変えることによって、費用が必要分だけで済むようになる。
- コストの最適化(データセンターの運用と保守への投資が不要に)
  　今までインフラに対してかけていたコストが、必要なくなる。
- キャパシティ予測が不要に
  　オンプレの場合は、アクセス数とかを予測してそれに合わせて最大のインフラ容量を用意しないといけないため、見積もりが難しかったり少ないアクセスの時に余計なコストがかかっていることになってしまう。クラウドであれば、都度スケーリングできるので予測の必要がなくなる。
- スケールによる大きなコストメリット
  アクセスに合わせてスケーリングするから、必要最低限のコストで済むことになる。
- スピードと俊敏性の向上
  　あらかじめAWS側でリソースが用意されているから、すぐにリソースを調達することができ、検証や開発にかかるコストを削減できる。
- わずか数分で世界中にデプロイ
  既にインフラが整えられているので、すぐにデプロイ（=公開）することができる。

6つ一応定義されてるけど、内容若干かぶってて3つに絞れそう。。

# AWS Well-Architectedフレームワーク

- オペレーショナルエクセレンス（Operational Excellence）
  PDCAみたいに、運用をうまく回して良くしていこうって感じ。
  ⇒あなたの「PDCAみたいに、運用をうまく回して良くしていこう」は方向性としてかなり良いです。
  ただし、公式寄りに言うと “運用する・監視する・知見を得る・改善する” 能力 です。単なる運用保守より、継続的改善の仕組み まで含みます。
- セキュリティ（Security）
  　文字通りシステムを守ろうって感じ。
  　⇒ただ、試験ではもう一段具体的に、情報・システム・資産の保護、リスク評価、緩和策、IAM、検知、データ保護、インシデント対応 あたりまでつながると強いです。
- 信頼性（Reliability）
  　どんな時も常に一定のサービスを提供しようねって感じ。
  　⇒「どんな時も常に一定のサービスを提供しよう」は悪くないですが、少し抽象的です。
  公式では、障害やサービス中断から回復できること、需要に応じて動的にリソースを確保できること、設定ミスや一時的なネットワーク問題にも対処できること が含まれます。
  つまり、単に安定していることだけでなく、壊れても戻れること が重要です。
- パフォーマンス効率（Performance Efficiency）
  　文字通り効率よく稼働させようって感じ。
  ⇒「効率よく稼働させよう」で方向性は合っています。
  ただ、公式ニュアンスは 要件を満たすために適切なコンピューティングリソースを効率よく選び、需要や技術進化に合わせて改善し続けること です。
  なので「省リソース」だけでなく、適材適所で選ぶこと も含みます。
- コスト最適化（Cost Optimization）
  　とにかく無駄をなくそうって感じ。
  ⇒「無駄をなくそう」は半分正解です。
  ただ、公式では ライフサイクル全体を通じて継続的に改善し、コストを最小化しつつ投資対効果を最大化すること がポイントです。
  つまり、単なる節約ではなく、必要な価値を出しつつ最適化する です。
- 持続可能性（Sustainability）
  　今のシステムをいつまでも継続できるようにって感じ。
  ⇒ここは一番修正したいです。
  あなたの説明だと「今のシステムをいつまでも継続できるように」に見えますが、AWS公式の Sustainability は環境持続性 の話です。特に エネルギー効率や資源使用量の削減など、環境への影響を下げること に重点があります。
  「サービスを長く続ける」という意味ではなく、環境負荷を下げる設計 と覚えるのが正確です。

# Design For Failure

- 単一障害点をなくそう。
  データセンターやハードウェアを一つにしないで複数にすることで、障害が起きてもカバーできるようにする。
  ⇒Design for Failure = “障害は必ず起こる前提で設計する” です。
  単にデータセンターやハードウェアを複数化するだけでなく、次も含みます。

- 単一障害点をなくす

- 障害時に自動復旧できるようにする

- フェイルオーバーできるようにする

- 影響範囲を小さくする

- 障害を前提にテストする

なので、あなたの説明は 一部として正しい ですが、
「冗長化」だけに寄っていて、「障害前提」「自動復旧」まで言えるともっと正確 です。

# コンポーネントの分離

- システムのコンポーネントを疎結合にする。
  コンポーネント間で過度に密接にしないことで、どこかが止まってもシステム全体が止まらないようにする仕組み

- マイクロサービスアーキテクチャ
  大規模システムを機能単位でサービスとして分割し、設計する手法
  ⇒これも概ねOKです。
  「大規模システムを機能単位で分割する」 という理解は合っています。

ただし注意点は、
マイクロサービス = 疎結合の実現方法の1つ であって、
疎結合そのものとイコールではない ことです。

つまり、

疎結合は設計の考え方

マイクロサービスはその考え方を実現する代表的なアーキテクチャ
